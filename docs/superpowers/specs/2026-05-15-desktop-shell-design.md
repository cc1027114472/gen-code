# Desktop Shell Design

**Date:** 2026-05-15

**Goal**

在当前 `gen-code` 项目中新增一个独立的桌面端子项目，使用 `Wails + Go + 前端` 搭建第一阶段桌面壳框架。该阶段只要求桌面应用可启动、前后端桥接可用、目录结构清晰，不接入现有 HTTP 服务和数据库能力。

**Scope**

本阶段包含：

- 新增独立 `desktop/` 子目录
- 配置 Wails 基础运行框架
- 提供一个最小 Go 绑定对象供前端调用
- 提供一个最小前端首页，验证桌面窗口与绑定调用
- 提供基本开发说明与运行入口

本阶段不包含：

- 复用现有 `cmd/server` 启动链路
- 接入数据库、用户模块或 worker
- 接入当前后端 HTTP 服务
- 实现正式业务页面或复杂状态管理

**Architecture**

桌面端作为仓库内的独立子项目存在，与现有 Go 服务骨架解耦。Wails 负责窗口生命周期和前后端桥接，Go 侧只暴露一个最小 `App` 绑定对象，前端负责展示启动状态和调用绑定结果。

第一阶段优先保证边界清晰，而不是过早共享代码。后续如果要接现有服务能力，可以再决定是通过本地函数绑定还是嵌入服务模块的方式整合。

**Directory Layout**

推荐目录结构如下：

```text
desktop/
  main.go
  app.go
  wails.json
  go.mod
  README.md
  frontend/
    package.json
    index.html
    src/
      main.(ts|js)
      App.(tsx|jsx)
      style.css
```

说明：

- `desktop/main.go`：Wails 启动入口，负责窗口配置和绑定注册
- `desktop/app.go`：最小 Go 绑定对象，提供 `Ping` 或 `GetAppInfo` 之类的方法
- `desktop/frontend/`：前端壳工程，负责渲染桌面首页并调用 Go 绑定
- `desktop/README.md`：记录开发、运行、构建方式

**Runtime Flow**

运行流程如下：

1. 启动 `desktop` 子项目的 Wails 应用
2. Wails 创建桌面窗口并加载前端资源
3. 前端页面初始化后调用 Go 绑定方法
4. Go 返回简单字符串或应用信息
5. 前端展示调用结果，作为桥接成功标志

**UI Requirements**

首页只承担“框架联通验证”职责，保持极简：

- 显示应用标题
- 显示桌面端已启动状态
- 提供一个按钮调用 Go 绑定方法
- 展示绑定方法返回值

不引入复杂视觉系统，不做业务模块导航。重点是确认桌面壳、前端资源加载和 Go 绑定三者全部打通。

**Error Handling**

- 如果前端调用 Go 绑定失败，页面需要展示明确错误信息
- 如果桌面应用启动配置错误，应保证错误可在开发终端直接看到
- 不为本阶段增加全局错误系统，只做最小可诊断输出

**Testing Strategy**

本阶段验证以运行性为主：

- Go 代码能够编译
- 前端依赖能够安装并启动
- Wails 应用能够打开窗口
- 前端能够成功调用至少一个 Go 绑定方法

如果仓库环境允许，可增加最小 Go 单测覆盖 `App` 绑定对象中的纯函数行为；但不为此引入额外复杂测试设施。

**Decisions**

- 技术栈固定为 `Wails + Go + 前端`
- 第一阶段选择“纯桌面壳”方案，不复用当前服务启动链路
- 桌面端放在独立 `desktop/` 子目录，而不是并入现有 `cmd/server`
- 优先建立清晰边界，后续再决定共享与整合方式

**Success Criteria**

完成后应满足以下结果：

- 仓库中存在可独立运行的 `desktop/` 子项目
- 桌面窗口能正常打开
- 前端首页能正常渲染
- 前端能调用 Go 绑定方法并展示结果
- 现有 `gen-code` 服务骨架不受影响
