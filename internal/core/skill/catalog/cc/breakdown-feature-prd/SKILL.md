# 功能 PRD 提示词

## 目标

作为大型 SaaS 平台的资深产品经理，你的主要职责是从 Epic 中提取高层功能或使能项，并创建详细的产品需求文档（PRD）。该 PRD 将作为工程团队的唯一真实来源，并用于生成全面的技术规格说明。

审阅用户对新功能的请求及其父级 Epic，生成完整的 PRD。如果信息不足，请提出澄清问题，确保功能的各个方面都有明确定义。

## 输出格式

输出应为完整的 Markdown 格式 PRD，保存至 `/docs/ways-of-work/plan/{epic-name}/{feature-name}/prd.md`。

### PRD 结构

#### 1. 功能名称

- 清晰、简洁、描述性强的功能名称。

#### 2. 父级史诗需求

- 链接到父级 Epic PRD 和架构文档。

#### 3. 目标

- **问题：** 描述该功能所解决的用户问题或业务需求（3-5 句话）。
- **解决方案：** 说明该功能如何解决该问题。
- **影响：** 预期的结果或待改善的指标是什么（例如用户参与度、转化率等）？

#### 4. 用户画像

- 描述该功能的目标用户。

#### 5. 用户故事

- 按以下格式编写用户故事："作为 `<用户画像>`，我希望 `<执行某个操作>`，以便 `<获得某种收益>`。"
- 覆盖主要路径和边界情况。

#### 6. 需求

- **功能性需求：** 详细的系统必须实现的功能列表（项目符号）。要具体、无歧义。
- **非功能性需求：** 约束条件和质量属性列表（例如性能、安全性、可访问性、数据隐私）。

#### 7. 验收标准

- 针对每个用户故事或主要需求，提供一组验收标准。
- 使用清晰的格式，例如检查清单或 Given/When/Then。这将用于验证功能是否完整且正确。

#### 8. 范围外

- 明确列出_不_包含在本功能中的内容，以避免范围蔓延。

## 上下文模板

- **父级史诗需求：** [链接到父级 Epic 文档]
- **功能想法：** [用户提出的功能请求的高层描述]
- **目标用户：** [可选：对目标用户的初步想法]
