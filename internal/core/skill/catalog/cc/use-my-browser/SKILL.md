# 使用我的浏览器

这是一个面向复杂网页任务的核心浏览器自动化策略 skill。它会教会 agent 何时继续停留在公共网页工具，何时切换到实时浏览器会话，以及何时回退到独立浏览器或原始抓取路径。

虽然名字叫“use my browser”，但它覆盖的不只是“使用我当前的浏览器”。它还教授更广义的浏览哲学：先定义目标，再选择合适的网络层；把结果当作证据；优先使用一手来源而不是转述；在有实时会话参与时尽量减少对用户的打扰。

## 何时使用

当以下任一情况成立时，使用这个 skill：

- 用户想做公共网页研究、来源校验或页面检查，而工具选择本身很重要
- 任务依赖用户当前的已登录状态或 cookies
- 用户已经在 Chrome DevTools 中打开了相关页面、元素或失败请求
- 目标是社交平台、强反爬站点或其他动态页面，静态抓取很可能拿不到真实内容
- 目标页面是动态的、需要认证的，或仅靠搜索与原始抓取难以检查
- 任务需要从实时浏览器会话中获取当前 DOM、控制台、网络活动、性能数据、文件上传过程或渲染后的媒体内容
- 用户希望 agent 接手他们已经在 Chrome 中手动开始的调试流程

在以下情况下不要使用这个 skill：

- 任务完全是本地工作，与网页无关
- 使用一个干净的独立浏览器上下文会比碰用户当前的实时浏览器更安全
- 用户明确要求不要使用他们当前的浏览器会话

## 预检查

在选择浏览层之前：

- 如果任务是公开的、并且强依赖引用，先从 `web` 开始
- 如果任务需要实时浏览器会话，先确认 Chrome DevTools 连接是可用的，再依赖它
- 如果 `references/site-patterns/` 下已有目标域名的说明，先读它
- 实时浏览器会话的回退与恢复规则见 [`references/session-playbook.md`](./references/session-playbook.md)

## 第一原则

### 1. 先定义成功，再选工具

从目标开始，而不是从工具开始。先明确什么才算完成：

- 用户真正需要的信息或动作是什么？
- 这个任务是否需要用户当前的浏览器状态？
- 最终答案更偏向引用密集、交互密集，还是调试密集？

### 2. 从最便宜且可能成功的层开始

使用成本最低、但仍有可能完成任务的那一层：

- 公共网页工具：用于发现、引用和普通页面读取
- 处理后读取或原始读取：用于更便宜地提取内容或获取源级响应数据
- 浏览器工具：用于依赖实时状态、交互或渲染证据的任务

详细路由规则见 [`references/tool-matrix.md`](./references/tool-matrix.md)。

### 3. 把每一个结果都当作证据

不要盲目重复同一种失败策略。每一步都应该更新计划：

- 搜索结果可能表明目标是公开的、缺失的，或藏在登录后
- 一次快照可能揭示数据其实已经在 DOM 里，不需要 OCR
- 找不到某个 DOM 节点，可能意味着页面懒加载、虚拟化、被交互门控，或者你压根就在错误的页面上
- 平台显示“not found”，可能意味着访问路径有问题，而不是真的不存在

### 4. 保护用户的会话

实时浏览器会话非常有价值，应谨慎使用：

- 当任务依赖当前状态时，优先复用已连接的 Chrome DevTools 会话
- 不要关闭或劫持你没有自己打开的标签页
- 除非任务的重点就是当前已选页面、元素或请求，否则优先使用你自己的工作标签页进行探索
- 离开时让会话比接手时更干净

### 5. 搜索用于发现来源，不用于证明结论

搜索引擎和聚合页是发现工具。当任务重点是真实性或验证时：

- 用搜索定位可能的来源
- 在做强结论前直接阅读来源
- 优先使用官方文档、官方页面、原始公告和一手内容，而不是被反复转述的摘要

### 6. 优先使用站点原生链接与完整参数

如果页面本身已经暴露了某个链接，应优先使用它提供的完整 URL，而不是手工拼接。查询参数、令牌和生成路径往往携带真实的会话或路由上下文。

## 会话边界与安全

Chrome DevTools MCP 的官方文档在这里很重要：

- 实时会话访问可能暴露所选 Chrome profile 中的所有打开窗口
- `--autoConnect` 是把真实 Chrome 会话分享给 MCP 客户端的现代安全方式
- `--browser-url` 是显式 remote-debug-port 配置下的回退方案，但不应推荐用于用户的默认浏览 profile
- 在默认 profile 上手动打开 remote debugging port，风险高于正常的实时浏览器会话流程，因为任何本地进程都可能在端口开放期间连接

把这些当作背景规则：

- 如果当前 DevTools 连接已经存在，优先使用它
- 只有在需要解释配置背景或排障时，才提到 `--autoConnect` 和 `--browser-url`
- 不要把用户导向 remote-debug-port 工作流，除非当前环境确实需要这种回退

## 核心工作流

### 目标优先的浏览循环

1. 定义成功条件
2. 选择最便宜且最有希望的层
3. 把结果当作证据进行检查
4. 只有当前层无法达到目标时才升级
5. 当目标达成时就停止，而不是把所有可能路径都试一遍

### 默认实时浏览器会话流程

当任务依赖用户当前的浏览器会话时：

1. 用 `list_pages` 查看当前有哪些页面
2. 如果用户当前页面就是任务本身，就复用该页面
3. 否则打开或切换到一个专门的工作页面再做探索
4. 如果结构化页面数据可能已经足够，先用 `take_snapshot` 再考虑截图
5. 把 `evaluate_script`、network tools、console tools 和 performance tools 作为主要证据来源
6. 只有在视觉状态本身是重点，或者 DOM 无法暴露足够信息时，才使用截图

具体浏览器操作与等价工具映射见 [`references/browser-recipes.md`](./references/browser-recipes.md)。

## Chrome DevTools 交接模式

### 从已选元素继续

如果用户已经在 Elements 面板中选中了某个元素：

- `take_snapshot` 可以暴露当前选择上下文
- `evaluate_script` 往往是下一步最好的工具，用来读取计算值、属性、状态或周围 DOM
- 只有在确认了当前结构后，再使用 `click`、`fill`、`hover` 等交互工具

### 从已选网络请求继续

如果用户已经在 Network 面板中选中了某个失败请求：

- 先调用不带 `reqid` 的 `get_network_request`
- 在查看更广泛的请求模式之前，先检查请求/响应体、header、状态、时间线和失败形状
- 只有在需要周边上下文或比较多个请求时，再使用 `list_network_requests`

### 从正在调试的页面继续

如果用户已经停留在问题页面上：

- 直接从该页面继续，而不是重新打开一个隔离副本
- 除非用户要求重新复现，否则尽量保留他们已经搭好的状态
- 如果你需要一个额外标签页进行安全实验，请自己创建，并保留原始页面不动

## 提取与交互规则

详细模式见 [`references/session-playbook.md`](./references/session-playbook.md)。简短版如下：

- 优先使用 DOM 与网络证据，而不是 OCR
- 在制定交互计划时，优先使用 `take_snapshot` 而不是 `take_screenshot`
- 当数据很可能存在但不可见时，优先使用 `evaluate_script`
- 根据站点真正会响应的方式，在直接提取与 GUI 风格交互之间切换
- 把截图、手工拼出的 URL 和“not found”页面都当作需要验证的对象，而不是直接当真

## 公共网页 vs 实时会话

这个 skill 不应该退化成“总是用浏览器”。

- 当真正的问题是“应该先选哪一层”时，即便是公共网页任务也属于这个 skill
- 对于强依赖引用的工作，先从公共网页工具开始，只有在更便宜的路径无法达到目标时才升级
- 路由选择见 [`references/tool-matrix.md`](./references/tool-matrix.md)，具体浏览器操作见 [`references/browser-recipes.md`](./references/browser-recipes.md)

## 并行研究策略

对于多个彼此独立的公共研究目标：

- 在可能时把 `web.search_query` 批量放到一次调用中
- 读取多个来源时批量使用 `web.open`
- 对独立的 shell 或本地文档读取使用 `multi_tool_use.parallel`，不要把它用在依赖同一可变页面状态的浏览器步骤上
- 关于何时值得使用 agent 级并行，以及如何安全地组织，见 [`references/session-playbook.md`](./references/session-playbook.md)

## 红旗信号

如果出现以下任一情况，就应停止并切换方法：

- 你正打算为了一个只需要公共引用的任务去使用实时浏览器会话
- 你正打算关闭、刷新或导航用户仍可能正在使用的页面
- 你在检查 DOM 或网络是否已经有答案之前，就先去截图
- 你忽略了当前 DevTools 中已经选中的请求或元素，转而从头开始
- 页面明明已经暴露了真实链接和参数，你却要手工构造 URL
- 你在还没验证访问路径是否正确之前，就把平台的“not found”当成最终结论
- 你正打算建议在用户正常浏览 profile 上开启 remote debugging
- 任务明明依赖用户当前已登录的 Chrome 状态，你却习惯性使用 Playwright

## 示例

**应该使用这个 skill**

- “我已经在 DevTools 里选中了失败的请求。解释一下它为什么返回 403。”
- “检查这个已登录的仪表盘，不要让我再登录一次。”
- “我已经在 Elements 里点中了坏掉的组件。找出布局为什么不对。”
- “从这个懒加载页面里提取真实的图片或视频源。”
- “读取这个公开文档，告诉我结构化元数据；如果 fetch 已经足够，就不要打开浏览器。”
- “比较这三个公开来源，给出引用，只有在静态路径失败时才使用浏览器。”
- “去这个社交站点里找到真实内容，即使搜索结果和静态抓取都不可靠。”

**不要使用这个 skill**

- “重命名这些本地文件并更新 import 路径。”
- “重构这个解析器并运行单元测试。”
- “打开一个干净浏览器并测试未登录的注册流程。”
