# 工具矩阵

在判断任务应该停留在公共网页工具、切入实时浏览器会话，还是使用单独的自动化浏览器时，使用这张表。

| 场景 | 主要工具 | 原因 | 避免 |
| --- | --- | --- | --- |
| 公共发现、最新信息、需要引用支撑的回答 | `web.search_query` 加 `web.open` | 来源发现快，引用方便，不会侵入浏览器 | 只是为了读公共页面就强行接入浏览器会话 |
| 已知的公共页面，且渲染态并不重要 | `web.open` 或 `web.find` | 成本最低的读取路径 | 还没证明需要时就启动浏览器工具 |
| 已知文章、文档或 PDF，且更便宜的处理后读取已经足够 | `web.open` 或 Jina Reader 这类 markdown 镜像 | 适合内容优先页面，且 token 更省 | 对文章型页面一上来就开浏览器 |
| 需要原始 HTML、headers、JSON-LD 或源级元数据 | 使用原生抓取工具的 `shell_command` | 能保留原始响应形态 | 把处理后读取当成源 HTML 使用 |
| 任务依赖当前登录状态、cookies 或应用上下文 | `chrome-devtools` | 复用实时浏览器会话 | 除非用户明确要干净状态，否则不要直接换 Playwright |
| 用户已经在 Elements 中选中了元素 | `chrome-devtools.take_snapshot` 再 `evaluate_script` | 沿用现有调试上下文 | 默认在新浏览器里重建整个页面 |
| 用户已经在 Network 中选中了请求 | `chrome-devtools.get_network_request` | 直接读取当前请求 | 还没检查当前请求就把所有请求都列出来 |
| 动态 SPA、懒加载、控制台报错、网络时序、性能、上传 | `chrome-devtools` | 对实时 DOM、网络和性能证据覆盖最好 | 使用会错过应用状态的原始抓取 |
| 社交平台或强反爬站点，静态抓取效果差 | `chrome-devtools` | 能在真实渲染页面和真实会话上下文中工作 | 假设搜索结果或静态抓取就能代表真实页面 |
| 需要单独的干净浏览器上下文，或可复现的隔离自动化 | `playwright` | 最适合隔离自动化流程 | 当目标是状态隔离时，继续复用实时会话 |
| 已在页面中发现真实 URL，需要下载公开资源 | 使用原生抓取工具的 `shell_command` | 比对着截图分析一个可直接抓取的文件更合适 | 用截图处理原始抓取问题 |
| 需要检查或比较多个公共来源 | 批量 `web` 调用 | 支持并行，且便于引用 | 为每个来源都串行接入浏览器 |
| 需要读取多个彼此独立的本地文档或 shell 结果 | `multi_tool_use.parallel` | 可干净地并行开发者工具读取 | 把它拿去做 `web` 操作或依赖同一浏览器状态的动作 |
| 用户明确要求并行 agent 处理多个独立浏览目标 | 目标驱动的委派 worker 加独立页面上下文 | 让各目标互相隔离，不打扰同一个可变页面状态 | 把依赖同一可变页面状态的工作硬拆开 |

## 决策规则

### 在以下情况下从 `web` 开始

- 回答需要引用公共来源
- 页面不是用户私有的
- 搜索质量比交互更重要
- 任务可以从文档、博客、发布说明或公共页面解决
- 第一问题是来源发现，而不是页面交互

### 在以下情况下从处理后读取开始

- URL 已知，且页面主要是文章、文档或 PDF 内容
- 你要的是页面文本，而不是原始 HTML
- token 效率重要，且页面结构并不高度交互

### 在以下情况下从 `chrome-devtools` 开始

- 任务依赖用户当前 Chrome 状态
- 相关元素或请求已经在 DevTools 中选中
- 目标在登录后，或者依赖当前应用会话
- 你需要从真实页面获取 DOM、控制台、网络、性能、上传或渲染媒体
- 站点动态性强到静态抓取并不可靠

### 在以下情况下从 `playwright` 开始

- 用户需要干净、独立的浏览器上下文
- 任务是一个不应触碰实时会话的浏览器自动化流程
- 当前环境没有接上所需的 Chrome DevTools 会话

### 在以下情况下从原始抓取开始

- 你需要 HTML 源码、响应 headers 或直接文件下载
- 浏览器渲染本身并不是关注点
- 任务核心是响应体，而不是页面中的行为
- 你已经发现了准确的资源或页面 URL，并且需要它的源级形式

## 升级顺序

当第一层失败时，要有节制地升级：

1. `web`
2. 处理后读取
3. 原始抓取
4. `chrome-devtools`
5. `playwright`

这不是一条僵硬的阶梯。如果任务从一开始就明显依赖登录态或 DevTools 上下文，就直接从 `chrome-devtools` 开始。

## 说明

- `web` 已经支持在一次调用里批量发起多个搜索或打开请求。
- `multi_tool_use.parallel` 只用于开发者工具，不适用于 `web` 命名空间。
- 当公共、低风险的路径已经足够时，不要使用实时浏览器会话。
- 当页面已经暴露真实链接时，优先使用站点生成的 URL，而不是手工猜测出来的 URL。
