# 会话操作手册

本手册记录了详细模式，用来让实时浏览器会话真正有用，同时避免变得过于侵入。

## 1. 有意识地复用当前会话

当任务依赖以下内容时，优先使用实时 Chrome DevTools 会话：

- 已登录状态
- 当前 cookies 或应用上下文
- 已经在 Elements 或 Network 中选中的目标
- 代价高、很难重新复现的现有应用状态

先检查当前已经打开了什么：

- 用 `list_pages` 了解有哪些页面可用
- 用 `select_page` 切换到正确页面
- 在交互前先用 `take_snapshot` 检查当前结构

当用户的当前状态本身就是任务重点时，直接使用用户现有页面。否则，创建或选择你自己的工作页面，以免打扰他们的主标签页。

如果预期应该有实时浏览器会话，但当前不可用：

- 清楚说明你目前没有实时浏览器会话
- 只有在公共网页路径或 Playwright 路径仍能服务目标时，才回退到它们
- 说明干净的回退浏览器并不等价于用户当前已登录的会话

## 2. 标签页卫生

### 优先采用非破坏性行为

- 不要关闭不是你自己打开的页面
- 不要因为方便就重新加载用户的页面
- 除非任务或用户明确需要，否则不要把页面切到前台
- 如果实验可能造成干扰，打开单独页面并保持原页面不变

### 何时直接使用当前活动页

在以下情况下直接使用当前活动页：

- 用户明确要求你从他们当前的调试状态继续
- 相关请求或元素已经被选中
- 在别处重现该状态会丢失任务的重点

## 3. DOM 优先提取

优先使用结构化证据，而不是截图：

1. `take_snapshot`：用于可访问页面结构和已选元素上下文
2. `evaluate_script`：用于数据、状态、属性、计算值和 DOM 遍历
3. Network 与 Console 工具：当答案可能并不直接显示在 DOM 中时使用
4. `take_screenshot`：只有当渲染后的像素本身真的重要时才使用

### 适合 DOM 优先的情况

- 已经挂载在 DOM 中的隐藏文本
- 链接目标和图片源地址
- 数据属性、ARIA 状态、class 和内联样式
- 像 `currentTime`、`duration`、暂停状态这样的 video 元数据
- 可通过脚本访问到的开放 shadow root 或同源页面结构中的值

### DOM 优先可能失败的情况

- 跨域 iframe 或不可访问的嵌入上下文
- 纯 canvas 或只能通过截图理解的内容
- 像素表现比 markup 更重要的视觉 bug

如果 DOM 路线卡住了，就切换到更合适的层，而不是强行 OCR 或反复快照。

### 站点生成的链接优于猜测 URL

如果页面本身已经暴露了链接、图片 URL 或媒体 URL：

- 优先使用站点生成的精确 URL
- 除非你明确知道 query string 和其他上下文参数是可选的，否则保留它们
- 如果站点开始返回误导性错误，就把“手工简化的 URL”视为可疑对象

## 4. GUI 交互 vs 程序化交互

有两种主要交互风格：

- 程序化：通过 `evaluate_script` 直接读写 DOM、快速导航和结构化提取
- 类 GUI：`click`、`fill`、`hover`、键盘输入、滚动和截图，更像真实用户行为

在以下情况下优先使用程序化方式：

- 数据本来就已经存在于页面中
- 你需要可靠提取，而不是模拟用户
- 直接访问 DOM 比重放 UI 步骤更简单

在以下情况下优先使用 GUI 风格交互：

- 真实用户手势本身很重要
- 站点对直接 DOM 操作反应很差
- 页面需要滚动、展开、悬停或上传交互，才能暴露真实状态

如果某个站点抗拒某一种模式，就切换，而不是继续重复同一种尝试。

如果 UI 看起来像阻塞，先判断它是否真的阻塞了数据。有时内容其实已经存在于 DOM 或网络层，屏幕上的交互只是表现形式。

## 5. 懒加载、隐藏数据与媒体

### 懒加载页面

- 在判定内容缺失前先滚动
- 重大状态变化后重新快照
- 在 resort 到截图之前，先寻找内容 URL、占位符或 hydration 数据
- 当访问路径或参数可能有误时，要谨慎看待“空白”或“not found”这类 UI 状态

### 图片与媒体

- 优先从 DOM 或页面脚本中提取图片和媒体 URL
- 只有当渲染后的图像本身就是分析对象时才使用截图
- 如果文件 URL 是公开且可直接下载的，优先原始抓取，而不是分析整页截图

### 视频采样

在相关 `<video>` 元素上使用 `evaluate_script` 来：

- 检查 `currentTime`、`duration` 和播放状态
- 在页面允许时 seek 到目标时间戳
- 暂停在有代表性的画面

如果需要做视觉分析，再用 `take_screenshot` 抓取渲染后的画面。

## 6. 登录处理

实时浏览器会话之所以有价值，是因为用户可能已经登录。

### 默认规则

先尝试通过实时浏览器会话访问目标。如果数据已经可用，就继续，不要把登录变成独立任务。

### 何时暂停并询问

只有在以下情况下才暂停并询问：

- 所需内容明显被认证阻挡
- 当前实时会话并没有提供访问权限
- 在用户真实 Chrome 中登录后，任务就能继续推进

此时，要求用户先在 Chrome 中登录，等他们确认后再继续。除非用户明确要求，否则不要默认在隔离浏览器中重现登录流程。

## 7. DevTools 交接

### 已选元素工作流

- 先从 `take_snapshot` 开始
- 再用 `evaluate_script` 检查附近的 DOM 状态、计算值或做定向提取
- 只有在理解了当前结构后，才使用交互工具

### 已选网络请求工作流

- 先调用不带 `reqid` 的 `get_network_request`
- 检查状态、headers、payload 和失败文本
- 只有当周边流量也很重要时，才扩大到 `list_network_requests`

## 8. 公共来源验证

当任务关注的是真相，而不是交互时：

- 优先一手来源，而不是二手报道
- 用 `web` 做发现和引用
- 只有当来源是动态的、需要登录，或公共网页路径无法访问时，才接入实时浏览器会话

浏览器并不会自动比公共网页路径更权威。要为你需要支撑的结论选择合适的层。

## 9. 故障排除式回退

如果实时浏览器会话路线被卡住：

- 先检查目标是否真的需要实时浏览器会话
- 检查更便宜的处理后读取或原始抓取路径是否已经足够回答问题
- 再考虑公共 `web` 路径是否其实已经够用
- 当任务需要独立、干净的浏览器上下文时，使用 Playwright
- 只有当问题确实和会话接入或 Chrome DevTools MCP 配置有关时，才提到 `--autoConnect` 或 `--browser-url`

## 10. 并行研究

当多个目标彼此独立时：

- 在动用 agent 之前，先批量发起 `web` 查询
- 为每个浏览器调查保持独立的页面上下文
- 如果用户明确要求并行 agent 工作，就围绕目标和成功条件来组织每个委派任务，而不是过早把工具层写死

不要拆分那些依赖同一个不断变化页面状态的工作。
