# 完成前验证

## 概述

在未验证的情况下声称工作已完成，不是高效，而是不诚实。

**核心原则：** 永远先有证据，再有声明。

**违背这条规则的字面要求，就是违背这条规则的精神。**

## 铁律

```
没有新鲜的验证证据，就不要声称已经完成
```

如果你没有在当前这条消息里运行验证命令，你就不能声称它通过了。

## 闸门函数

```
在声称任何状态或表达满意之前：

1. IDENTIFY：哪个命令能证明这个声明？
2. RUN：执行完整命令（重新运行、完整运行）
3. READ：阅读完整输出，检查退出码，统计失败数
4. VERIFY：输出是否支持这个声明？
   - 如果不支持：带证据说明真实状态
   - 如果支持：带证据做出声明
5. ONLY THEN：只有这时才能声明

跳过任何一步 = 不是在验证，而是在撒谎
```

## 常见失败

| 声明 | 需要什么 | 不足以证明 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 个失败 | 之前跑过、或者“应该能过” |
| Linter 干净 | Linter 输出：0 个错误 | 部分检查、主观外推 |
| 构建成功 | 构建命令：退出码 0 | Linter 通过、日志看起来不错 |
| Bug 已修复 | 原始症状的测试现在通过 | 代码变了、主观认为修好了 |
| 回归测试有效 | 已验证红-绿循环 | 测试只通过一次 |
| Agent 已完成 | VCS diff 显示有变更 | Agent 报告“成功” |
| 需求已满足 | 逐条核对清单 | 测试通过 |

## 危险信号——停下

- 使用 “should”、“probably”、“seems to” 之类措辞
- 在验证前就表达满意（“Great!”、“Perfect!”、“Done!” 等）
- 在未验证前就准备提交 / push / 创建 PR
- 相信 agent 的成功报告
- 依赖部分验证
- 想着“就这一次”
- 很累，只想赶紧结束
- **任何在尚未运行验证时暗示成功的措辞**

## 防止合理化

| 借口 | 现实 |
|--------|---------|
| “现在应该能工作了” | 去运行验证 |
| “我很有信心” | 信心 ≠ 证据 |
| “就这一次” | 没有例外 |
| “Linter 过了” | Linter ≠ 编译器 |
| “Agent 说成功了” | 要独立验证 |
| “我太累了” | 疲惫 ≠ 借口 |
| “部分检查已经够了” | 部分验证什么都证明不了 |
| “换个说法，这条规则就不适用了” | 精神重于字面 |

## 关键模式

**测试：**
```
✓ [运行测试命令] [看到：34/34 通过] “所有测试通过”
✗ “现在应该能过了” / “看起来没问题”
```

**回归测试（TDD 红-绿）：**
```
✓ 编写 → 运行（通过）→ 回退修复 → 运行（必须失败）→ 恢复 → 运行（通过）
✗ “我已经写了回归测试” （但没有做红-绿验证）
```

**构建：**
```
✓ [运行构建] [看到：退出码 0] “构建通过”
✗ “Linter 通过了” （linter 并不检查编译）
```

**需求：**
```
✓ 重读计划 → 创建清单 → 逐项核对 → 报告缺口或完成情况
✗ “测试通过了，所以这个阶段完成了”
```

**Agent 委派：**
```
✓ Agent 报告成功 → 检查 VCS diff → 验证变更 → 报告真实状态
✗ 直接相信 agent 的报告
```

## 为什么这很重要

来自 24 次失败记忆：
- 你的人类搭档说过“我不相信你”——信任被破坏
- 未定义函数被发出去了——会直接崩溃
- 缺失需求被发出去了——功能并不完整
- 时间浪费在虚假的“已完成”上 → 被打回 → 返工
- 这违反了：“诚实是核心价值。如果你撒谎，你就会被替换。”

## 何时应用

**始终在以下动作之前：**
- **任何**形式的成功/完成声明
- **任何**满意表达
- **任何**关于当前工作状态的正面表述
- 提交、创建 PR、标记任务完成
- 移动到下一个任务
- 把任务委派给 agents

**这条规则适用于：**
- 精确措辞
- 改写和同义表达
- 成功的暗示
- **任何**暗示完成/正确性的沟通

## 最终结论

**验证这件事，没有捷径。**

运行命令。阅读输出。然后再声明结果。

这不可协商。
