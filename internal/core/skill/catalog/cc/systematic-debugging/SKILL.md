# 系统化调试

## 概述

随机修补只会浪费时间，还会引入新的 bug。快速补丁会掩盖底层问题。

**核心原则：** 在尝试修复之前，始终先找到根因。只修症状就是失败。

**违背这个流程的字面要求，就是违背调试的精神。**

## 铁律

```
没有根因调查，就不要开始修复
```

如果你还没完成第 1 阶段，你就不能提出修复方案。

## 何时使用

适用于**任何**技术问题：
- 测试失败
- 生产环境中的 bug
- 异常行为
- 性能问题
- 构建失败
- 集成问题

**尤其在以下情况下更要使用：**
- 有时间压力时（紧急情况最容易让人靠猜）
- “先来个快速修复”看起来很显然
- 你已经尝试过多个修复
- 上一个修复没起作用
- 你并不完全理解这个问题

**即使在这些情况下也不要跳过：**
- 问题看起来很简单（简单 bug 也有根因）
- 你很赶时间（匆忙只会保证返工）
- 经理要你“现在立刻修好”（系统化比瞎撞更快）

## 四个阶段

在进入下一阶段前，你**必须**完成当前阶段。

### 阶段 1：根因调查

**在尝试任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们常常就包含精确解法
   - 完整阅读堆栈跟踪
   - 记下行号、文件路径、错误码

2. **稳定复现**
   - 你能稳定触发它吗？
   - 具体步骤是什么？
   - 是每次都会发生吗？
   - 如果无法复现 → 收集更多数据，不要猜

3. **检查最近改动**
   - 有什么变化可能导致这个问题？
   - `git diff`、最近的提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，先加诊断埋点：**
   ```
   对每一个组件边界：
     - 记录进入组件的数据
     - 记录离开组件的数据
     - 验证环境/配置是否被正确传递
     - 检查每一层的状态

   先运行一次，收集证据，看清楚它是在哪里坏掉的
   然后再分析证据，定位失败的组件
   然后再调查那个具体组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的 secrets：==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量：==="
   env | grep IDENTITY || echo "IDENTITY not in environment"

   # 第 3 层：签名脚本
   echo "=== Keychain 状态：==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这能揭示：** 是哪一层出了问题（secrets → workflow ✓，workflow → build ✓，……）

5. **追踪数据流**

   **当错误出现在调用栈很深的位置时：**

   查看当前目录下的 `root-cause-tracing.md`，了解完整的逆向追踪技术。

   **简化版：**
   - 这个错误值最初来自哪里？
   - 是谁用错误值调用了这里？
   - 一直向上追，直到找到源头
   - 在源头修，而不是在症状处修

### 阶段 2：模式分析

**修复前先找模式：**

1. **寻找正常工作的例子**
   - 在同一代码库里定位相似且正常工作的代码
   - 有什么工作正常的东西和当前坏掉的东西很相似？

2. **对照参考实现**
   - 如果你正在实现某种模式，要**完整**阅读参考实现
   - 不要略读——每一行都读
   - 在应用之前彻底理解这个模式

3. **识别差异**
   - 正常工作的和坏掉的之间有什么不同？
   - 把每个差异都列出来，不管看起来多小
   - 不要假设“这个不可能有影响”

4. **理解依赖**
   - 它还依赖哪些其他组件？
   - 需要哪些设置、配置、环境？
   - 它基于什么前提假设？

### 阶段 3：假设与测试

**科学方法：**

1. **建立单一假设**
   - 明确写出：“我认为 X 是根因，因为 Y”
   - 把它写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做**尽可能小**的改动来验证假设
   - 一次只改一个变量
   - 不要一次修多个东西

3. **继续之前先验证**
   - 起作用了吗？是 → 进入第 4 阶段
   - 没起作用？建立**新的**假设
   - 不要在上一次失败的修复上继续叠补丁

4. **当你不知道时**
   - 直接说“我不理解 X”
   - 不要假装懂
   - 求助
   - 继续研究

### 阶段 4：实施修复

**修根因，不修症状：**

1. **创建失败的测试用例**
   - 尽可能简单地复现
   - 能自动化就自动化
   - 没有框架就写一次性测试脚本
   - 修复前**必须**有这个测试
   - 使用 `superpowers:test-driven-development` skill 来编写合格的失败测试

2. **实施单一修复**
   - 针对已识别出的根因
   - 一次**只做一个**改动
   - 不要“顺手”改进其他东西
   - 不要把重构打包在一起

3. **验证修复**
   - 测试现在通过了吗？
   - 有没有别的测试坏掉？
   - 问题真的解决了吗？

4. **如果修复没起作用**
   - 停下
   - 数一下：你已经尝试了多少次修复？
   - 如果 < 3：回到第 1 阶段，带着新信息重新分析
   - **如果 ≥ 3：停下并质疑架构（见下面第 5 步）**
   - 在没有做架构讨论前，不要尝试第 4 次修复

5. **如果 3 次以上修复都失败：质疑架构**

   **表明存在架构问题的模式：**
   - 每次修复都会在不同地方暴露新的共享状态 / 耦合 / 问题
   - 修复需要“巨大重构”才能落地
   - 每次修复都会在别处制造新症状

   **停下来，质疑基本前提：**
   - 这个模式在根本上是否合理？
   - 我们是不是“只是靠惯性硬撑着继续用它”？
   - 与其继续修症状，我们是不是该重构架构？

   **在尝试更多修复之前，先和你的人类搭档讨论**

   这**不是**一次失败的假设，而是错误的架构。

## 危险信号——停下并回到流程

如果你发现自己在想：
- “先快速修一下，之后再调查”
- “先改改 X 试试有没有用”
- “一起加几个改动，然后跑测试”
- “先不写测试，我手动验证一下”
- “大概率就是 X，我先修它”
- “我没完全理解，但这个也许能行”
- “模式是这么说的，但我稍微改造一下”
- “这里的主要问题是：[列出修复方案而不是调查结果]”
- 在追踪数据流之前就开始提解决方案
- **“再试一个修复”**（已经试了 2 次以上时）
- **每次修复都在不同位置引出新问题**

**以上全部意味着：停下。回到第 1 阶段。**

**如果 3 次以上修复失败：** 质疑架构（见第 4.5 阶段）

## 你的人类搭档发出的“你做错了”信号

**注意这些纠偏语句：**
- “那件事不是没发生吗？”——你在没验证的情况下做了假设
- “这样能让我们看到……吗？”——你本该加入证据采集
- “别猜了”——你在没理解前就提修复方案
- “Ultrathink this”——要质疑根本前提，而不只是症状
- “我们卡住了吗？”（带挫败感）——你的方法不奏效

**一旦看到这些：** 停下。回到第 1 阶段。

## 常见合理化借口

| 借口 | 现实 |
|--------|---------|
| “问题很简单，不需要流程” | 简单问题也有根因。流程对简单 bug 同样很快。 |
| “这是紧急情况，没时间走流程” | 系统化调试比猜来猜去更快。 |
| “先试这个，再调查” | 第一次修复就决定了后续模式。从一开始就做对。 |
| “先确认修好了，再写测试” | 未经测试的修复站不住。先写测试才能证明它。 |
| “一次修多个更省时间” | 你无法隔离到底什么起作用，还会引入新 bug。 |
| “参考实现太长了，我自己改造下模式” | 理解不完整一定会出 bug。完整读完。 |
| “我看出问题了，我来修” | 看见症状 ≠ 理解根因。 |
| “再尝试一次修复” （2+ 次失败后） | 3+ 次失败 = 架构问题。质疑模式，不要继续修。 |

## 速查表

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、复现、检查改动、收集证据 | 理解 WHAT 和 WHY |
| **2. 模式** | 找正常案例、做对比 | 识别差异 |
| **3. 假设** | 建立理论、最小化测试 | 假设被确认或产生新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当流程得出“没有根因”

如果系统化调查表明问题确实是环境问题、时序依赖问题或外部问题：

1. 你已经完成了这个流程
2. 记录你调查了什么
3. 实施合适的处理方式（重试、超时、错误信息）
4. 为未来调查加入监控/日志

**但是：** 95% 的“没有根因”其实都是调查不完整。

## 支持技术

这些技术属于系统化调试的一部分，并可在当前目录中找到：

- **`root-cause-tracing.md`** - 沿调用栈向后追踪 bug，找到最初触发点
- **`defense-in-depth.md`** - 在找到根因后，在多层加上校验

**相关 skills：**

- **superpowers:verification-before-completion** - 在声称成功前验证修复确实生效

## 现实影响

基于调试会话的经验：
- 系统化方法：15-30 分钟修复
- 随机修补方法：2-3 小时混乱试错
- 一次修好率：95% vs 40%
- 引入新 bug：几乎为零 vs 经常发生
