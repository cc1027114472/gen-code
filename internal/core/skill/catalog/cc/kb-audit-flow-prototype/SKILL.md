# KB Audit 流程原型

## 概览

当任务是把 `kb-audit` 文档集转化为阶段性流程原型、按业务链扩展这些原型，或验证原型是否仍与文档中的业务流程一致时，在这个项目里使用此 skill。

这个 skill 覆盖完整的工作闭环：

- 从 `kb-audit` 文档重建上下文
- 识别目标链路和当前阶段
- 更新阶段性原型
- 校验原型与文档是否一致

在修改任何内容之前，先阅读 `references/source-of-truth.md`。

## 何时使用

当用户提出以下任何需求时，使用这个 skill：

- 继续推进这个项目的阶段性原型
- 添加劳务、营收侧、机械、审批或其他文档中已有的链路
- 检查当前原型是否与业务流程文档一致
- 修正菜单、页面、节点顺序与业务含义之间的不一致
- 构建或修订从 `kb-audit` 派生出来的 HTML 演示原型

以下场景不要使用这个 skill：

- 这个项目之外的通用前端工作
- 仅做 Axure 原生实现
- 无关的落地页或营销站点

## 核心规则

### 1. 文档是真值来源

不要只从当前原型出发。

始终先从相关文档重建上下文：

- 目标链路对应的业务流程总览文档
- 站点菜单手册或产品模块理解文档
- 只要涉及审批，就必须阅读审批中心设计文档

### 2. 按链路、按阶段构建

默认扩展顺序：

1. 首页与导航
2. 采购加审批
3. 劳务
4. 营收侧
5. 机械

对于每条链路：

- 保留文档中定义的完整链路
- 让当前阶段具备交互能力
- 在需要时让后续节点或较轻量节点保持可见
- 明确标注哪些部分是完整交互，哪些只是上下文展示

### 3. 保持原型与菜单对齐

如果某个节点有页面，就也应该有可发现的入口，除非明确有意不暴露它。

同时检查以下三层：

- 业务链路节点
- 左侧导航或页面入口
- 实际渲染出来的页面实现

除非用户明确要求保留占位，否则不要留下“有页面但没有菜单入口”或“有节点标签但没有页面”的缺口。

### 4. 审批必须表现为控制层

当目标链路包含审批时：

- 把审批展示为控制层，而不是孤立的装饰物
- 让审批动作在概念上始终依附于业务单据
- 让强绑定规则清晰可见，例如“审批通过前不能生成订单”

### 5. 每次重大修改后都要校验

始终执行以下检查：

- 对 JS 代码做语法检查
- 确认相关文件存在
- 将原型中的链路文本与文档链路进行比对
- 记录任何有意的范围压缩或交互深度不足之处

## 输出默认值

默认原型输出目录：

- `F:\codex\artifacts\kb-audit\html-demo-prototype`

默认文件：

- 页面文件 `index.html`
- 样式文件 `styles.css`
- 脚本文件 `app.js`
- 使用说明 markdown 文件

除非用户明确要求单独做一个变体，否则就在现有阶段性原型上原地更新。
