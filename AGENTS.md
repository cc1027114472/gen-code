# AGENTS.md

## 编码规则

1. 每写一个功能，必须先考虑如何测试，并补齐一整条链路的测试；不能只做局部小测试就结束。
2. 新增或修改功能时，单元测试只作为辅助，最终必须完成从入口到出口的完整链路验证。
3. 如果功能涉及页面、浏览器交互、前后端联调或需要模拟真实用户操作，可以使用 `agent-browser` / 浏览器自动化来完成链路测试。
4. 代码变更必须与测试结论一起交付；如果某一环节无法测试，必须在说明里明确原因、缺口和风险。
5. 尽量沿用现有项目结构、命名和风格，不做无关重构。
6. 变更前先理解上下文，变更后检查是否影响现有测试和调用链。
7. 任何涉及行为变化的修改，都要说明影响范围和回归风险。
8. 最终目标是把 Codex 和 CC 可用的全部工具与 MCP 集成进来；如果工具能力有重复，优先选择最好的一套使用，不要求强行消除所有重复。
9. `skills` 必须分组隔离使用，不要混为一谈；通过分组选项切换使用 Codex 和 CC 各自的 `skills`，未来新增更多组时也要保持隔离。
10. 公共 `skills` 可以跨组复用，例如后续安装的通用 `skills`。
11. 当把 `skills` 复制到本项目中时，必须 1:1 汉化成中文，不要自行改写含义、重组内容或二次创作。
12. API 规范白名单只允许三种：`OpenAI Responses`、`Anthropic`、`Gemini`。
13. 明确不支持或不新增其他接口规范，包括但不限于 `OpenAI Compatible`、`Amazon Bedrock` 及其他兼容层或供应商私有变体。
14. 如果集成一个工具，必须把该工具声明支持的核心能力全部测试到位，不能只验证其中一小部分就视为集成完成。
15. 如果集成的是浏览器类工具，必须把该工具支持的各类浏览器使用方式逐项覆盖验证，例如打开页面、导航、点击、输入、截图、读取页面状态、处理交互等能力。
16. 开发过程中允许并鼓励使用多个子代理并行完成任务，以提升分析、实现、测试和验证效率。
17. 使用多子代理时，必须明确各自负责的模块、文件范围或任务边界，避免重复劳动和相互覆盖。

## 执行要求

- 默认以 `go test ./...` 作为基础验证，但它不能替代链路测试。
- 如果功能涉及 HTTP 接口、路由、中间件、配置、启动流程、页面交互或外部系统联动，必须补链路级验证。
- 当链路测试需要真实浏览器行为时，优先使用 `agent-browser` 或等价浏览器自动化方式。
- 如果测试失败，先修复问题再结束，不要只给出代码不闭环。
- 涉及模型接入、接口适配或供应商抽象时，只能围绕 `OpenAI Responses`、`Anthropic`、`Gemini` 三种规范设计。
- 涉及工具集成时，测试结果必须覆盖能力清单；要明确哪些能力已验证、如何验证、还有哪些边界未覆盖。
- 使用多子代理时，最终结果必须由主流程统一集成、复核和测试，不能把未收敛的并行结果直接作为最终交付。

## 项目空间与线程规则

1. `gen-code` 的目标形态是：先打开一个项目，创建一个本地空间（workspace），再在该空间内创建多个线程（thread / session）并行工作。
2. 多个线程必须彼此隔离、互不影响，至少包括：各自的会话上下文、消息历史、工具调用记录、运行状态、任务状态与中间输出。
3. 多个线程允许并发进行，但并发不等于共享运行时上下文；任何线程级状态都不能默认串线。
4. 多个线程共享同一个项目空间内的文档读取能力；项目文档、说明文件、共享知识材料默认可被多个线程共同读取。
5. “共享空间文档” 不等于 “共享线程内部状态”；共享的是工作区文档与公共资料，不是会话私有上下文。
6. 任何涉及线程写入共享文件、修改工作区文件、生成产物覆盖、工具副作用冲突的能力，都必须显式定义隔离策略、冲突策略和权限策略，不能隐式处理。
7. 在功能尚未完整实现前，所有设计、编码、测试和文档都必须以这个目标模型为准，避免把桌面壳、CLI、app-server 或 core 设计成单线程单上下文绑定结构。
